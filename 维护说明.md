# 维护说明

本文面向以后继续修改、排查问题、发布稳定版时使用。

## 维护原则

1. **小步修改**：一次只改一个主题，例如「footY 逻辑」「多 Lmi」「贴图工具」。
2. **先自动测试，再游戏实测**：自动测试通过只能证明工具内部流程没坏；最终仍要进 Hero Fighter / HF-EX 验证。
3. **保留可回退点**：每次稳定后提交 Git commit；大功能单独开分支或至少单独 commit。
4. **不要手工格式化 JSON 样本**：HFWorkshop JSON 的键序和数字原文很重要，数据兼容性靠 `jsonio.js` 保持。

## 建议的 Git 工作流

### 第一次初始化后

```bash
git status
git add .
git commit -m "chore: initialize HF action editor repository"
```

### 日常修改

```bash
git status
# 修改代码
node test/all.js
node test/browser/e2e.js
git add <改过的文件>
git commit -m "fix: describe the change"
```

### 推荐提交类型

- `feat:` 新功能，例如添加新的编辑工具。
- `fix:` 修 bug，例如修复某个角色显示异常。
- `test:` 新增或修改测试。
- `docs:` 文档更新。
- `chore:` 仓库结构、依赖、忽略规则等维护。

## 新角色异常排查流程

### 1. 打开后空白或缺部位

先看是否有缺部位弹窗。若弹窗列出类似：

- `Lucas_07LeftFist`
- `rudolf_00Head`
- `icefire`
- `weapon_effect_white`

说明 Spt 引用了其他 Lmi 的 Limb。处理方式：

1. 在 HFWorkshop 里搜索或导出包含这些 Limb 的 `*Lmi`。
2. 解压到当前角色目录旁边。
3. 重新打开角色。

工具会自动读取同目录下所有 `*Lmi` 文件夹。

### 2. 工具里正常，游戏里异常

优先检查：

- 是否导入了正确的 Spt zip 与对应 Lmi zip。
- 是否忘记导入被修改过的附加 Lmi zip。
- 是否修改了 footY、rootDx、ULseparate、ref 复用帧等敏感字段。
- 是否只编辑了复用帧的判定框，而期待姿势也变化。

### 3. 某个部位移动/旋转异常

优先检查：

- 当前「联动」开关是开还是关。
- 父部位是否缺失。
- 该槽位是否是从属同步部位（例如拳覆盖、护腕、鞋等）。
- 当前帧是否是 `refIndex` 复用帧（复用帧姿势只读）。

## footY 维护说明

`rebake.js` 中 footY 使用增量模式：

```text
新 footY = 原存档 footY + (编辑后公式值 - 编辑前公式值)
```

原因：HFEX 部分角色的存档 footY 与游戏公式存在系统性偏差。如果用公式绝对值覆盖，会让角色在游戏里悬空或下沉。

只有「帧」面板中的「重算」按钮应该做绝对重算，因为那是用户显式要求。

## 多 Lmi 维护说明

每个 Lmi 文件夹都是一个独立图池：

- `LimbPic_0.json` 在不同 Lmi 中可以同时存在。
- `0.png` 在不同 Lmi 中也可以同时存在。
- 因此代码里附加 Lmi 的 PNG 键使用 `LmiFolder/0.png` 形式区分。

不要把多个 Lmi 的 `picFiles` 直接合并到一个 `Map<number, JsonFile>`，否则图池编号会冲突。
